# 实验六：Block Nested Loop Join

## 1. 实验目的

本实验在 PostgreSQL 12.0 查询执行器中实现 Block Nested Loop Join。普通 Nested Loop Join 对外表中的每一条记录都会重新扫描一次内表；Block Nested Loop Join 一次缓存一批外表元组，并用这批元组共同扫描一次内表，从而减少内表重复扫描次数。

本实验使用 `block_nested_loop_size` 控制外表块大小，并测试 `1、2、8、64、128、1024` 六种取值。

## 2. 修改内容

本次主要修改以下源码文件：

- `src/backend/executor/nodeNestloop.c`
- `src/include/nodes/execnodes.h`
- `src/backend/utils/misc/guc.c`
- `src/backend/optimizer/path/costsize.c`
- `src/include/optimizer/cost.h`

### 2.1 执行器状态扩展

在 `NestLoopState` 中增加外表 block 相关状态：

- `nl_BlockJoin`：当前 Nested Loop 节点是否启用 block join。
- `nl_BlockSize`：本次执行使用的 block size。
- `nl_BlockCount`：当前 block 中实际缓存的外表 tuple 数量。
- `nl_BlockIndex`：当前正在和内表 tuple 匹配的外表 tuple 下标。
- `nl_OuterExhausted`：记录外表是否已经到达 EOF，避免最后一个 block 处理结束后再次调用外表扫描并导致重复扫描。
- `nl_CurrentInnerTupleSlot`：保存当前内表 tuple，保证一次 `ExecProcNode` 返回后下次调用能从正确位置继续处理当前内表 tuple。
- `nl_OuterBlockSlots`：用于保存外表 tuple 副本的 `TupleTableSlot` 数组。

### 2.2 执行流程

原始 Nested Loop Join 的执行逻辑是：

1. 读取一个外表 tuple。
2. 从头扫描内表。
3. 对每个内表 tuple 判断 join 条件。
4. 外表 tuple 处理完后读取下一个外表 tuple，并重新扫描内表。

修改后的 Block Nested Loop Join 执行逻辑是：

1. 从外表读取最多 `block_nested_loop_size` 个 tuple，并复制到 `nl_OuterBlockSlots`。
2. 从头扫描内表。
3. 对每个内表 tuple，依次与 block 内的所有外表 tuple 判断 join 条件。
4. 当前 block 与内表匹配结束后，再读取下一批外表 tuple。

当 `block_nested_loop_size = 1` 时，执行结果等价于普通 Nested Loop Join。实验实现只对无外部参数依赖的 `INNER JOIN` 启用 block join，其它 join 类型仍使用 PostgreSQL 原始逻辑。

### 2.3 GUC 参数

新增参数：

```sql
SET block_nested_loop_size = 64;
```

该参数为 `PGC_USERSET`，默认值为 `1`，允许范围为 `1` 到 `1024`。因此可以在同一个 PostgreSQL 服务中通过 SQL 直接切换 block size。

## 3. 编译与运行

编译安装：

```sh
./build_postgres.sh
```

运行实验：

```sh
./run_experiment.sh
```

实验脚本会完成以下工作：

1. 初始化并启动当前目录下的 `pgsql/data`。
2. 创建 `similarity` 数据库。
3. 导入 `similarity_data.sql`。
4. 关闭 hash join 和 merge join，强制使用 Nested Loop。
5. 分别设置六种 block size，运行两次查询并记录第二次结果。

测试 SQL：

```sql
SELECT count(*)
FROM restaurantaddress ra, restaurantphone rp
WHERE ra.name = rp.name;
```

## 4. 实验结果

结果文件为 `block_performance.txt`。表中 `second_ms` 为第二次执行耗时，`shared_read` 为第二次执行中读取的 shared blocks 数。

| block size | first ms | second ms | shared hit | shared read |
| --- | ---: | ---: | ---: | ---: |
| 1 | 1513.545 | 1482.719 | 46 | 0 |
| 2 | 1008.650 | 993.241 | 46 | 0 |
| 8 | 630.213 | 636.656 | 46 | 0 |
| 64 | 511.605 | 510.768 | 46 | 0 |
| 128 | 503.447 | 524.601 | 46 | 0 |
| 1024 | 501.915 | 519.906 | 46 | 0 |

正确性使用 `validate_block_join.sh` 中的小数据集验证。`block_nested_loop_size` 分别为 `1、2、8` 时，连接结果数量均应为 `5`，且执行计划包含 `Nested Loop`。

验证输出摘要如下：

```text
block=1 | 5
block=2 | 5
block=8 | 5
Nested Loop
```

## 5. 结果分析

Block Nested Loop Join 的核心收益来自减少内表扫描次数。普通 Nested Loop Join 的内表扫描次数约等于外表 tuple 数；block size 为 `B` 时，内表扫描次数约减少为 `ceil(outer_rows / B)`。

实验结果符合预期：`block_nested_loop_size = 1` 时第二次执行耗时约 `1482.719 ms`，增大到 `64` 后降到约 `510.768 ms`。继续增大到 `128` 和 `1024` 后收益不明显，原因是内表重复扫描次数已经较少，剩余开销主要来自 tuple 复制、join 条件判断和执行器循环本身。

## 6. 总结

本实验通过扩展 PostgreSQL 12.0 的 Nested Loop executor，实现了可配置 block size 的 Block Nested Loop Join。该优化不改变 join 语义，只改变执行顺序和性能特征。通过 GUC 参数可以在运行时切换 block size，便于比较不同 block size 下的执行性能。
